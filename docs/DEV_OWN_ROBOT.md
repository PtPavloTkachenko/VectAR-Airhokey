# Dev note — owning / reviving an OSKR robot (signing + BLE flash)

Not part of the product flow (a real OSKR owner already has their key; a stock
owner uses the firmware install). This is for **our** dev unit, whose own SSH
key was regenerated by a Clear User Data wipe, and captures what's verified so a
5-minute finish is possible once the one offline dependency is back.

## The state we're in

Our unit runs DDL `2.0.1.6091oskr` (**ankidev**). After the wipe its
`/data/ssh/authorized_keys` was rebuilt by `localsshuser.sh` to trust exactly:
- its **own** regenerated key `id_rsa_Vector-XXXX` (now `Vector-X1W8`) — only
  obtainable from its log bundle (~149k BLE packets ≈ a day; RtsLogRequest's
  Mode/Filter are unimplemented in firmware, so it can't be narrowed).
- the `digital_dream_labs_dev_key` (fp `SHA256:XBuEGYmhtwUicEOx…`) — DDL's dev
  key, whose **private half was never published**.

So no key we hold currently opens it. The robot is **locked out, not corrupted**
— which is why backups don't fix it: backups restore *state we have*, but the
missing thing is a *write channel we don't have*.

## The write channels that exist (and why each is blocked)

| Channel | Can it install a key we control? |
|---|---|
| SSH | circular — that's what we're trying to get |
| adb `:5555` | closed on this unit |
| BLE `RtsSshRequest` | in the CLAD schema but **not implemented** by the firmware (send succeeds, no reply) |
| BLE log download | read-only, and full-bundle only (~a day) |
| BLE `OTAStart` | **works** — but it flashes the *system* partition, which needs a signed image |

`OTAStart` is the lever. It rewrites the OS, so we can put an image that trusts a
key we hold — the "custom OSKR OS" path.

## What's VERIFIED for the OTAStart path

1. **Our robot's OTA trust anchor is the community `ota_prod.key`.**
   `wire-os/ota/ota_prod.pub` **verifies the signature of our own known-good
   `vicos-2.0.1.6091oskr.ota`** → `openssl dgst -sha256 -verify ota_prod.pub
   -signature manifest.sha256 manifest.ini` → **"Verified OK"**. So an OTA whose
   `manifest.sha256` is signed with `ota_prod.key` will be accepted (die 209 is
   the safe failure if it isn't).
2. **wire-os authorizes a key we hold.** Its `localsshuser.sh` adds three keys;
   one is `victor-dev@anki` = `SHA256:9lZMgxdfKD9A…` = the `ssh_root_key` from
   `kercre123/unlocking-vector` (saved at `~/.vectar/ssh_root_key`).
3. **The robot accepts ankidev images over BLE.** Verified live: OTAStart drove
   the robot to fetch the image from our Mac; the ep image was rejected only by
   the **build-type** gate (die 214: ankidev OS can't install a non-ankidev
   image), not the transport.

## The build recipe (from `froggitti/signman` + wire-os `ota/Makefile`)

An OTA tar = `manifest.ini`, `manifest.sha256`, `apq8009-robot-boot.img.gz`,
`apq8009-robot-sysfs.img.gz`. Images are `aes-256-ctr -md md5 -pass file:ota.pas`
(`ota.pas` = the well-known dev passphrase `這是一個密碼`). The signature is:
```
openssl dgst -sha256 -sign ota_prod.key -out manifest.sha256 manifest.ini
```
The wire-os `dev.ota` (WireOS 3.0.1.32d, ankidev=1) ships **already-encrypted
images but NO `manifest.sha256`** (it targets the no-signing bootloader). So for
our signature-checking robot we only need to **sign its `manifest.ini` and add
`manifest.sha256`** — no re-encrypt.

## The one blocker

A usable **`ota_prod.key`**:
- unencrypted copy lived at `api.froggitti.net/ota_p2.key|ota_prod.key` — the
  host is **currently down**, and not in the Wayback Machine.
- the copy committed in `wire-os/ota/ota_prod.key` is **passphrase-encrypted**;
  the passphrase is a build arg (`build.sh -op …`), not in git, and none of the
  obvious candidates (`這是一個密碼`, empty, wire/anki/vector…) open it.

Everything else is staged (`~/.vectar/ota/dev.ota`, `~/.vectar/ssh_root_key`,
`onboarding/flash_ota.py`). When the key is in hand: sign `dev.ota`'s manifest,
serve it, `flash_ota` over BLE, reboot, `ssh -i ~/.vectar/ssh_root_key … root@…`,
then `oskr_provision`. Then optionally flash our `6091oskr` back from backup.

## Tooling added this session
- `onboarding/flash_ota.py` — serve an OTA + drive `OTAStart` over BLE with a
  Wi-Fi join and handshake retry (also the mechanism behind the product's
  stock-firmware install).
- `onboarding/get_robot_key.py` / `probe_logs.py` — BLE log-bundle key recovery
  (works; slow) and a Mode/Filter probe (proved the firmware ignores filters).
- `onboarding/restore_ssh.py` — the RtsSshRequest attempt, kept with a loud note
